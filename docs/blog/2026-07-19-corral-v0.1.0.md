# Corral v0.1.0 — a portable Proxmox for people who already have a cluster (or just a laptop)

I love Proxmox. I've run it for years. But Proxmox is a *distribution* — it owns
the whole machine, it's welded to Debian, and if your infrastructure has moved
on to Kubernetes, it's a second world you have to maintain next to the first.

Corral is my answer to a question that kept nagging me: **what if the Proxmox
experience — the datacenter tree, the create wizard, one-click consoles, VMs
and containers side by side — was just a single static binary you point at
whatever you already have?**

- Got a Kubernetes cluster? Corral drives KubeVirt through `kubectl` and
  `virtctl` — no operator to install, no agent, no CRDs of its own.
- Just a laptop? The same commands run VMs on local QEMU/KVM under systemd.
- Got Tailscale? Every VM lands on your tailnet automatically — SSH from your
  phone, VNC from the couch.

One binary. `create` / `start` / `ssh` / `viewer` / `clone` / `delete` work
identically on both backends, and Corral remembers which VM lives where. There
is a TUI for quick jobs, a Proxmox-style web dashboard for everything else,
and a Proxmox API compatibility layer if your Terraform provider expects one.

Today's v0.1.0 is the first versioned release, and it ships the feature that
changed how I develop the thing itself: **demo mode**.

## Try it in 30 seconds, literally no cluster

```bash
curl -fsSL https://raw.githubusercontent.com/tuna-os/corral/main/scripts/install.sh | sh
corral --demo
```

Or via Homebrew:

```bash
brew install hanthor/tap/corral
```

To be clear about what you just installed: **that one binary is the whole
product** — the CLI, the TUI (`corral` with no arguments), *and* the web
dashboard (`corral web` serves it; there's no separate web-UI package,
container, or frontend build to deploy).

`--demo` boots an in-memory fake cluster *inside* the binary: three nodes,
eight VMs in every state you'd meet in real life (running, stopped, paused,
mid-install, a paused Windows desktop, an ephemeral scratch VM with a TTL
counting down), two containers, live CPU metrics. It's not a mockup — the
real CLI, TUI, and web UI run their actual code paths against it, and the
state is live: stop a VM in the dashboard and watch the TUI agree.

I built it so I could polish the interfaces without burning a cluster, and it
turned out to be the best onboarding tool Corral has. Kick every tire —
start, stop, clone, bulk-select, tag filters, the create wizard — before you
touch a single real machine.

## Walkthrough

### The TUI

Run `corral` bare (add `--demo` if you have nothing connected yet):

```
   Corral  ·  8 VMs  ·  2 CTs  ·  d: cluster health

▶ web-prod
  ● Running  kubevirt  ports:○  2 CPU / 4Gi  corral-1  10.42.1.20
  db-prod
  ● Running  kubevirt  ports:○  4 CPU / 8Gi  corral-2  10.42.2.31
  win11-desktop
  ⏸ Paused  kubevirt  ports:○  4 CPU / 8Gi  corral-2  10.42.2.55
  [CT] dev-shell
  CT  Running  1 CPU / 512Mi  privileged
```

Pick a VM, hit Enter, and you get an action menu that only offers what's
actually possible from the VM's current state — a running VM offers
Stop/Restart/Pause/Migrate; a stopped one offers Start/Clone/Snapshot.
Hit `d` for a cluster health screen that tells you exactly what your cluster
is missing and offers to fix the safe parts itself.

### The web dashboard

```bash
corral web            # binds 127.0.0.1:8006 — Proxmox's port, naturally
corral web --addr "$(tailscale ip -4):8006"   # ...or to your whole tailnet
```

You get the Proxmox shape you know: datacenter → node → guest tree on the
left, tabs on the right (Summary, Console, Terminal, Hardware, Options,
Snapshots, Events, YAML). Live CPU sparklines, bulk start/stop, tags with
tree filtering, one-click live migration with a target-node picker, disk
export, and real consoles in the browser — noVNC graphics and an xterm.js
serial terminal, both **vendored into the binary**, so they work in an
air-gapped homelab where a CDN `<script>` tag would silently strand you.

It's mobile-friendly, and since the binary is the server, "install the
management UI" means running one command.

### Creating things

```bash
# From the curated catalog (Fedora, Ubuntu, Debian, Rocky, openSUSE, ...)
corral create web --image fedora

# From an installer ISO — CDI imports it, progress shows in `corral list`
corral create win --kubevirt --iso https://.../installer.iso

# From a bootable container image — the OS disk is built *on the cluster*
corral plugin install bootc
corral create dev --bootc ghcr.io/tuna-os/yellowfin:gnome --wait-ssh
```

That last one is the trick I'm proudest of: point Corral at a bootc image and
it runs `bootc install to-disk` in a builder VM on the cluster, then boots the
result as a first-class VM. Your OS is a container image in a registry;
`corral bootc upgrade` moves the VM to the next build. And as of this release
`create --bootc` picks the backend for you — cluster if reachable, local QEMU
if not.

Containers get the Proxmox treatment too: `corral ct create` makes a "pet
pod" — a plain Kubernetes pod with a persistent volume and an init process,
presented like a Proxmox CT. In privileged mode it seeds a full root
filesystem onto the volume, distrobox-style, so `apt install` survives a
restart.

### When something's missing

```bash
corral doctor
```

Doctor now understands that *no cluster* is a valid way to run Corral: on a
bare laptop it prints one line explaining cluster checks are skipped, then
checks what actually matters locally — QEMU present, `/dev/kvm` usable,
Tailscale available. With a cluster connected it walks KubeVirt, CDI, feature
gates, storage classes, snapshot classes, and metrics-server, and
`corral doctor --fix` applies the safe, config-only repairs itself.

## What's next

v0.1.0 is where Corral stops being a science project and starts being a
product. The near-term list: a recorded demo for this README, deeper
devcontainer support for CTs, VDI desktop pools, and — once enough of you
star the repo to satisfy their notability bar — a formula in homebrew-core
proper.

If you run VMs on Kubernetes and miss the way Proxmox *feels*, or you run
Proxmox and wish it were one binary instead of an operating system: give
`corral --demo` thirty seconds. That's the pitch.

*Corral is Apache-2.0, single-binary Go, at
[github.com/tuna-os/corral](https://github.com/tuna-os/corral).*
